# GBServer自动化测试需求规格

## 1. 概述

### 1.1 项目背景
- **项目名称**: wvp-gb28181-server (GBServer)
- **技术栈**: Rust + Axum + SQLx + Redis
- **项目描述**: GB28181视频监控服务器后端(Rust重写)
- **测试目标**: 建立完整的自动化测试体系，确保系统稳定性、功能正确性和性能可靠性

### 1.2 测试范围
本测试方案覆盖以下核心模块：
- Web服务层（Axum框架的HTTP API）
- 数据库操作层（PostgreSQL/MySQL）
- SIP/GB28181协议栈
- ZLMediaKit集成模块
- Redis缓存模块
- JWT认证模块

### 1.3 测试类型
- 单元测试（Unit Tests）
- 集成测试（Integration Tests）
- 端到端测试（End-to-End Tests）
- 性能测试（Performance Tests）

---

## 2. 功能需求

### 2.1 单元测试需求

#### 2.1.1 核心业务逻辑单元测试
**需求描述**: 系统应为核心业务逻辑提供全面的单元测试覆盖。

**验收标准**:
- **[EARS-001]** 对于每个数据库操作模块（device、alarm、platform、user等），系统应提供独立的单元测试，验证CRUD操作的正确性。
- **[EARS-002]** 对于SIP协议核心模块（message、header、parser、transaction），系统应提供单元测试，验证协议解析和构建的正确性。
- **[EARS-003]** 对于GB28181业务模块（catalog、invite、ptz、cascade），系统应提供单元测试，验证国标协议实现的正确性。
- **[EARS-004]** 对于认证模块（auth.rs），系统应提供单元测试，验证JWT令牌生成、验证和刷新逻辑的正确性。
- **[EARS-005]** 对于缓存模块（cache.rs），系统应提供单元测试，验证Redis缓存读写、过期和失效逻辑的正确性。

#### 2.1.2 工具函数和辅助模块测试
**需求描述**: 系统应为工具函数和辅助模块提供单元测试。

**验收标准**:
- **[EARS-006]** 对于配置加载模块（config.rs），系统应提供单元测试，验证配置文件解析、默认值处理和环境变量覆盖的正确性。
- **[EARS-007]** 对于错误处理模块（error.rs），系统应提供单元测试，验证错误类型转换和错误消息格式化的正确性。
- **[EARS-008]** 对于响应构建模块（response.rs），系统应提供单元测试，验证API响应格式和状态码的正确性。

### 2.2 集成测试需求

#### 2.2.1 Web API集成测试
**需求描述**: 系统应为Web API提供集成测试，验证HTTP请求处理的完整流程。

**验收标准**:
- **[EARS-009]** 对于设备管理API（/api/device/*），系统应提供集成测试，验证设备注册、查询、更新和删除的完整流程。
- **[EARS-010]** 对于平台级联API（/api/platform/*），系统应提供集成测试，验证平台注册、通道同步和级联控制的完整流程。
- **[EARS-011]** 对于用户管理API（/api/user/*），系统应提供集成测试，验证用户创建、权限分配和认证授权的完整流程。
- **[EARS-012]** 对于媒体服务API（/api/media/*），系统应提供集成测试，验证直播、回放和下载的完整流程。
- **[EARS-013]** 对于告警管理API（/api/alarm/*），系统应提供集成测试，验证告警上报、查询和处理的完整流程。

#### 2.2.2 数据库集成测试
**需求描述**: 系统应为数据库操作提供集成测试，验证SQL语句执行和数据持久化的正确性。

**验收标准**:
- **[EARS-014]** 对于PostgreSQL数据库，系统应提供集成测试，验证所有SQL语句在真实数据库环境中的执行正确性。
- **[EARS-015]** 对于MySQL数据库，系统应提供集成测试，验证所有SQL语句在真实数据库环境中的执行正确性。
- **[EARS-016]** 对于数据库连接池，系统应提供集成测试，验证连接池创建、获取和回收的正确性。
- **[EARS-017]** 对于数据库事务，系统应提供集成测试，验证事务提交、回滚和并发控制的正确性。

#### 2.2.3 SIP协议集成测试
**需求描述**: 系统应为SIP协议栈提供集成测试，验证协议交互的完整流程。

**验收标准**:
- **[EARS-018]** 对于SIP UDP传输，系统应提供集成测试，验证SIP消息的发送、接收和路由的正确性。
- **[EARS-019]** 对于SIP TCP传输，系统应提供集成测试，验证SIP消息的发送、接收和路由的正确性。
- **[EARS-020]** 对于GB28181设备注册流程，系统应提供集成测试，验证设备注册、心跳保活和注销的完整流程。
- **[EARS-021]** 对于GB28181目录查询流程，系统应提供集成测试，验证目录订阅、通知和查询的完整流程。

#### 2.2.4 ZLMediaKit集成测试
**需求描述**: 系统应为ZLMediaKit集成提供集成测试，验证媒体服务交互的正确性。

**验收标准**:
- **[EARS-022]** 对于ZLM HTTP API调用，系统应提供集成测试，验证流媒体创建、查询和删除的正确性。
- **[EARS-023]** 对于ZLM Hook回调处理，系统应提供集成测试，验证鉴权、流注册和流注销的正确性。
- **[EARS-024]** 对于ZLM负载均衡选择，系统应提供集成测试，验证多节点选择和负载统计的正确性。

#### 2.2.5 Redis缓存集成测试
**需求描述**: 系统应为Redis缓存提供集成测试，验证缓存交互的正确性。

**验收标准**:
- **[EARS-025]** 对于Redis连接，系统应提供集成测试，验证连接建立、断线重连和连接池管理的正确性。
- **[EARS-026]** 对于缓存数据操作，系统应提供集成测试，验证缓存读写、过期设置和批量操作的正确性。
- **[EARS-027]** 对于缓存失效场景，系统应提供集成测试，验证缓存穿透、缓存击穿和缓存雪崩的处理正确性。

### 2.3 端到端测试需求

#### 2.3.1 完整业务流程测试
**需求描述**: 系统应提供端到端测试，验证完整业务流程的正确性。

**验收标准**:
- **[EARS-028]** 对于设备接入并播放视频流程，系统应提供端到端测试，验证从设备注册到视频播放的完整流程。
- **[EARS-029]** 对于平台级联并同步通道流程，系统应提供端到端测试，验证从平台注册到通道同步的完整流程。
- **[EARS-030]** 对于云台控制流程，系统应提供端到端测试，验证从API调用到设备响应的完整流程。
- **[EARS-031]** 对于录像回放流程，系统应提供端到端测试，验证从回放请求到流媒体传输的完整流程。

#### 2.3.2 WebSocket通信测试
**需求描述**: 系统应为WebSocket通信提供端到端测试。

**验收标准**:
- **[EARS-032]** 对于WebSocket连接建立，系统应提供端到端测试，验证连接握手、认证和会话管理的正确性。
- **[EARS-033]** 对于WebSocket消息推送，系统应提供端到端测试，验证设备状态推送、告警推送和流状态推送的正确性。

### 2.4 性能测试需求

#### 2.4.1 并发性能测试
**需求描述**: 系统应提供性能测试，验证系统在高并发场景下的表现。

**验收标准**:
- **[EARS-034]** 对于HTTP API并发请求，系统应提供性能测试，验证在1000并发请求下的响应时间和成功率。
- **[EARS-035]** 对于SIP消息并发处理，系统应提供性能测试，验证在500并发SIP消息下的处理能力和消息丢失率。
- **[EARS-036]** 对于数据库并发操作，系统应提供性能测试，验证在100并发数据库操作下的响应时间和死锁情况。

#### 2.4.2 压力测试
**需求描述**: 系统应提供压力测试，验证系统的极限性能和稳定性。

**验收标准**:
- **[EARS-037]** 对于设备注册压力测试，系统应验证在10000设备同时注册时的系统稳定性和响应时间。
- **[EARS-038]** 对于视频流并发压力测试，系统应验证在100路视频流同时播放时的系统资源占用和稳定性。
- **[EARS-039]** 对于长时间运行稳定性测试，系统应验证在连续运行24小时后的内存泄漏和资源释放情况。

## 3. 非功能需求

### 3.1 测试框架需求

#### 3.1.1 测试框架选择
**需求描述**: 系统应选择合适的Rust测试框架和工具链。

**验收标准**:
- **[EARS-040]** 系统应使用Rust内置测试框架（cargo test）作为主要测试工具。
- **[EARS-041]** 系统应使用tokio::test宏支持异步测试。
- **[EARS-042]** 系统应使用mockall库支持Mock对象生成。
- **[EARS-043]** 系统应使用criterion库支持性能基准测试。

#### 3.1.2 测试辅助工具
**需求描述**: 系统应配置测试辅助工具和库。

**验收标准**:
- **[EARS-044]** 系统应使用testcontainers库提供Docker容器化的测试环境（数据库、Redis等）。
- **[EARS-045]** 系统应使用wiremock库提供HTTP Mock服务。
- **[EARS-046]** 系统应使用assert_matches库提供模式匹配断言。
- **[EARS-047]** 系统应使用pretty_assertions库提供可读性更好的断言输出。

### 3.2 测试数据管理需求

#### 3.2.1 测试数据准备
**需求描述**: 系统应提供测试数据准备机制。

**验收标准**:
- **[EARS-048]** 系统应提供测试数据fixtures，支持测试数据的快速初始化。
- **[EARS-049]** 系统应提供测试数据生成器，支持随机测试数据的生成。
- **[EARS-050]** 系统应提供数据库迁移脚本，支持测试数据库的快速搭建。

#### 3.2.2 测试数据隔离
**需求描述**: 系统应保证测试数据的隔离性。

**验收标准**:
- **[EARS-051]** 每个测试用例应使用独立的测试数据库，避免数据污染。
- **[EARS-052]** 每个测试用例应在测试完成后清理测试数据，保证测试环境的干净。
- **[EARS-053]** 并发测试应使用不同的数据集，避免数据竞争。

### 3.3 测试报告需求

#### 3.3.1 测试覆盖率报告
**需求描述**: 系统应生成测试覆盖率报告。

**验收标准**:
- **[EARS-054]** 系统应使用cargo-tarpaulin或cargo-llvm-cov生成代码覆盖率报告。
- **[EARS-055]** 系统应生成HTML格式的覆盖率报告，支持在浏览器中查看。
- **[EARS-056]** 系统的代码覆盖率目标应达到80%以上。

#### 3.3.2 测试结果报告
**需求描述**: 系统应生成详细的测试结果报告。

**验收标准**:
- **[EARS-057]** 系统应生成JUnit格式的测试报告，支持CI/CD集成。
- **[EARS-058]** 系统应提供测试执行时间统计，识别慢测试用例。
- **[EARS-059]** 系统应提供测试失败原因分析，便于问题定位。

### 3.4 CI/CD集成需求

#### 3.4.1 持续集成测试
**需求描述**: 系统应集成到CI/CD流程中。

**验收标准**:
- **[EARS-060]** 系统应在每次代码提交时自动运行单元测试和集成测试。
- **[EARS-061]** 系统应在Pull Request时自动运行测试，并阻止测试失败的代码合并。
- **[EARS-062]** 系统应定期运行端到端测试和性能测试（如每日构建）。

#### 3.4.2 测试环境管理
**需求描述**: 系统应提供测试环境的自动化管理。

**验收标准**:
- **[EARS-063]** 系统应使用Docker Compose或Kubernetes管理测试环境。
- **[EARS-064]** 系统应在测试开始前自动启动依赖服务（数据库、Redis、ZLM等）。
- **[EARS-065]** 系统应在测试结束后自动清理测试环境。

## 4. 测试组织需求

### 4.1 测试目录结构
**需求描述**: 系统应组织清晰的测试目录结构。

**验收标准**:
- **[EARS-066]** 单元测试应放置在源文件同目录的tests模块中（#[cfg(test)] mod tests）。
- **[EARS-067]** 集成测试应放置在项目根目录的tests目录中。
- **[EARS-068]** 测试辅助代码应放置在tests/common目录中。
- **[EARS-069]** 测试数据应放置在tests/fixtures目录中。

### 4.2 测试命名规范
**需求描述**: 系统应遵循统一的测试命名规范。

**验收标准**:
- **[EARS-070]** 测试函数命名应遵循test_<被测功能>_<测试场景>_<预期结果>的格式。
- **[EARS-071]** 测试模块命名应与被测模块命名一致。
- **[EARS-072]** 测试文件命名应与被测文件命名一致。

## 5. 排除范围

以下内容不在本次自动化测试范围内：
- 前端界面测试（前端复用web目录，不在Rust后端测试范围）
- 第三方库内部逻辑测试（信任第三方库的测试）
- 生产环境数据测试（仅使用测试数据）
- 硬件设备物理连接测试（仅测试软件逻辑）

---

## 6. 假设与约束

### 6.1 假设
- 假设开发团队熟悉Rust测试框架和工具链
- 假设CI/CD环境支持Docker容器运行
- 假设有足够的计算资源运行性能测试

### 6.2 约束
- 测试执行时间应控制在合理范围内（单元测试<5分钟，集成测试<30分钟）
- 测试不应依赖外部网络资源（使用Mock服务）
- 测试不应破坏生产环境数据
